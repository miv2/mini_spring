## 저장소 가이드라인

## 프로젝트 구조 및 모듈 구성
소스 코드는 src/main/java/co/kr/mini_spring 아래에 위치하며, domain(JPA 엔터티/리포지토리), feature(인증/회원/게시글 컨트롤러 및 서비스), global(보안, 설정, 공통 응답)로 구분됩니다. 통합 테스트는 src/test/java 내에서 동일한 레이아웃을 따릅니다.

## 빌드, 테스트 및 개발 명령어
./gradlew clean build – 의존성을 해결하고, 단위/통합 테스트를 실행한 후 build/libs에 실행 가능한 JAR 파일을 생성합니다.

./gradlew bootRun – 현재 application.yml 설정으로 Spring Boot 애플리케이션을 시작합니다.

./gradlew test – MockMvc를 주로 사용하는 테스트 스위트만 실행합니다. API를 반복적으로 수정할 때 사용하세요.
Gradle 래퍼는 이미 커밋되어 있습니다. 버전 일관성을 유지하기 위해 시스템 Gradle을 직접 호출하는 것은 피하세요.

## 코딩 스타일 및 명명 규칙
Java 17과 4-공백 들여쓰기를 사용하세요. 클래스는 co.kr.mini_spring 네임스페이스에 유지하며, 클래스와 레코드에는 PascalCase, 메서드와 필드에는 camelCase, 열거형과 ResponseCode 상수에는 UPPER_SNAKE_CASE를 사용하세요. REST 컨트롤러는 가볍게 유지하고, 비즈니스 로직은 feature.*Service 클래스에 위임하세요. 생성자 주입(Lombok의 @RequiredArgsConstructor)과 소유한 기능 패키지 내의 재사용 가능한 DTO를 선호하세요. SQL 코드 조각이나 스키마 파일을 추가할 때는 src/main/resources 아래에 보관하세요.
기능구현시 각 메서드에 주석을 달아주세요. 엔드포인트 추가시 스웨거랑 API_DOCS.md 파일 최신화 해주세요

## 테스트 가이드라인
테스트는 Spring Boot Test + MockMvc에 의존합니다. 클래스 이름에는 *Test 접미사를 사용하고(예: AuthControllerTest), 관련 데이터 빌더는 해당 기능 패키지 옆에 배치하세요. 영속성 로직을 수정할 때는 정상 경로와 유효성 검사 오류를 모두 다루는 슬라이스 테스트나 MockMvc 흐름을 추가하세요. 실제 운영 데이터베이스에 직접 접근하지 마세요: 테스트 스위트를 실행하기 전에 application-test.yml이나 환경 변수를 통해 데이터소스 속성을 재정의하세요.

## 커밋 및 풀 리퀘스트 가이드라인
명령형 현재 시제를 사용하여 커밋 제목을 72자 이내로 작성하세요(예: fix: 리프레시 토큰 만료 처리). 관련 변경 사항은 커밋별로 그룹화하고, 중요하지 않은 변경의 경우 본문에 동기를 설명하세요. 풀 리큐스트는 변경 세트를 요약하고, 관련 이슈를 연결하며, 테스트 증거(./gradlew test 출력)를 상세히 기록하고, 설정이나 스키마 업데이트를 명시하여 검토자가 환경을 확인할 수 있도록 해야 합니다.

## 보안 및 구성 팁
실제 데이터베이스 자격 증명이나 JWT 비밀 키를 커밋하지 마세요. VCS에서 제외된 환경 변수나 .env 파일에 의존하세요. OAuth 클라이언트 ID는 보안 설정 저장소에 보관하고, 인증 스택을 업데이트할 때는 새로운 필수 변수를 HELP.md에 문서화하세요.
